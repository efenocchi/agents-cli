Verdict — is it messy? Not structurally. The architecture is clean and consistent: a single commander program, a uniform register<X>Command() convention, well-separated lib/ subsystems, a clear shared-core gravity (state/versions/agents/types), and it typechecks with zero errors plus 231 test files. But your instinct about duplication is correct and then some. Because the code grew area-by-area, the same low-level behaviors were re-implemented independently dozens of times — formatters, atomic writes, tilde expansion, version comparison, JSON parsing, process-liveness checks. The danger isn't aesthetics: several copies have silently diverged, and the divergent ones are bugs today (e.g. a buggy second copy of extractClaudeUserText, three NaN-broken compareVersions that drive the upgrade gate and prune-keeper selection, four formatBytes that disagree on the same file size). Reliability-wise the biggest theme is non-atomic config writes + read-side error-swallowing forming a corrupt-on-crash → silent-wipe loop over irreplaceable user state. The fixes are cheap and highly leveraged: ~5 shared helpers collapse the majority of findings at the source.

2 · Duplication findings

There is no shared src/lib/format.ts and no command framework — each area re-implements primitives. Below: the clusters that matter, HIGH = a copy has already drifted into a bug.

High extractClaudeUserText — buggy second copy

Two copies; the active.ts one is missing the isMeta / isLocalCommandMessage / [Request interrupted guards, so it emits wrong topic text (meta lines, command wrappers).

src/lib/session/active.ts:188  vs  src/lib/session/discover.ts:1775

Fix: export discover.ts's version, import it in active.ts.
High compareVersions — 3 copies, all NaN-broken on prerelease

Number('0-dev') → NaN, every NaN comparison is false → versions sort equal. This repo's own dev builds are 0.0.0-dev.<sha>. Drives the upgrade gate (index.ts:787), @latest/@oldest resolution, and prune keeper selection (view.ts:1261 — can move the wrong version to trash).

src/index.ts:242  ·  src/lib/versions.ts:1503  ·  src/commands/view.ts:141

Fix: one shared real-semver comparator in lib/.
High formatBytes — 4 implementations that disagree on the same file

sessions stops at MB (no GB), prune uses 2 decimals, inspect does KB/MB/GB/TB, browser handles undefined. A 1 GB file prints differently in three commands.

sessions.ts:143 · inspect.ts:385 (most complete) · prune.ts:197 · browser.ts:1838

Fix: one formatBytes in a new src/lib/format.ts based on inspect.ts's version.
High truncate() — 12 definitions, 2 are bugs

No shared truncate. cloud.ts:33 appends '...' (3 chars — overflows the width budget); sessions.ts:1560 appends a single '.' (reads as a sentence end). The other 9 correctly use '…' — a 9-vs-2 split confirms the odd two are bugs.

cloud.ts:33 · sessions.ts:1560 · cost.ts:187 · +9 more (teams, mcp, inspect, …)

Fix: one exported truncate(s,max) using '…'; delete all 12.
High formatAge — 4 copies, 4 different input contracts

Same name takes seconds (tmux), epoch-ms+"ago" (browser-picker), epoch-ms but drops the minutes remainder (browser.ts:1829 → "3h ago" never "3h 12m ago"), and ms-duration (pty).

tmux.ts:446 · browser-picker.ts:36 · browser.ts:1829 · pty.ts:435

Fix: split into formatAge(epochMs) and formatCompactDuration(ms) in the shared format lib.
High Inline process.kill(pid,0) liveness checks bypass canonical isAlive()

Canonical platform/process.ts:37 reimplemented 5×; the active.ts copy's EPERM handling is actually more correct and should be upstreamed.

platform/process.ts:37 (canonical) · session/db.ts:281 · session/active.ts:90 · runner.ts:470 · teams/agents.ts:824 · browser/runtime-state.ts:169

3 · Reliability findings

Dominant theme: non-atomic config writes + read-side error-swallowing = corrupt-on-crash → silent-wipe of irreplaceable user state. Two shared helpers close most of it.

High Non-atomic config writes everywhere → corruption on crash

Every config writer does read-modify-write directly onto the live file (permissions.ts, hooks.ts, shims.ts over .claude.json/settings.json, migrate.ts over agents.yaml, browser task state). A crash/SIGINT/full-disk mid-write truncates the file.

permissions.ts:1019,1069,1119,1200,1339 · hooks.ts:1078,1203,1430,1602 · shims.ts:1087,1182,1989 · migrate.ts:303,363,1477 · browser/service.ts:2162,2208

Fix: one shared atomicWriteJson(path,obj) = write path+'.tmp' then renameSync. Replace all sites. (Closes most of §3.)
High Read side swallows parse errors → indistinguishable from "file absent" → silent wipe

permissions.ts:873-951 return null on JSON.parse failure; a corrupted settings.json then reads as "no permissions," and applyClaudePermissions({merge:true}) finds nothing to merge and replaces. parseHookManifest (hooks.ts:784,802) swallows manifest errors → one bad indent in agents.yaml makes all hooks vanish, then the GC loop de-registers every managed hook.

Fix: propagate a distinct error on parse failure; never let a downstream merge/GC run against a file that failed to parse.
High mergeSqliteDb hard-deletes the source DB on ANY failure — irreplaceable session loss

migrate.ts:1085-1092 — bare catch swallows every failure (lock, disk full, SQL error) then fs.unlinkSync(src). A failed ATTACH-merge means rows may not have landed in dest. Compounded by SQL injection via string-interpolated ATTACH at migrate.ts:1039.

Fix: on merge failure, quarantine src to .history/trash/ instead of unlinking; parameterize the ATTACH.
High Non-atomic symlink retarget can destroy ~/.{agent}

shims.ts:933-935 — unlinkSync then mkdirSync then symlinkSync; between unlink and symlink the dir doesn't exist (the hot path runs agents sync constantly), and if symlinkSync throws the dir is gone with no rollback. switchHomeFileSymlinks already does this atomically; two siblings don't.

shims.ts:887,995,1132

Fix: shared atomicRetargetSymlink (temp-symlink + rename) used by all three.
High Subprocess calls without timeout → CLI hangs indefinitely
  • agents.ts:1190,1241,1265 — mcp list/add/remove no timeout (a hung mcp list stalls batch ops across N agents).
  • versions.ts:933,948,1198,1528 — npm view/npm install/--version no timeout.
  • discover.ts:1101-1148 — both openclaw execs no timeout (vs OpenCode's 5000ms).
  • sessions.ts:827-853 — spawn(resume) no watchdog and ignores the exit code (() => resolve() discards code).
Fix: add timeout to all execFile/exec calls; capture and propagate child exit codes.
Med Large mixed-responsibility functions (refactor candidates)
  • service.ts:309-438 start() (~130 lines) — profile lookup + health-check + unbounded fork spin-wait (while(forkingProfiles.has) await sleep(50), no deadline) + window + state + tabs.
  • permissions.ts:1163-1382 applyPermissionsToVersion (~220 lines) — 7 agents' convert+parse+merge+2 writes in one try; partial success leaves config mutated but returns {success:false}.
  • shims.ts:251-565 generateShimScript (~315 lines of embedded bash, untestable); migrate.ts:1634-1690 runMigration — ~30 sequential migrators, one async reject strands later repairs.
Fix: extract sub-steps; add a deadline to the fork spin-wait; isolate each migrator (try/catch per step) so one failure doesn't strand the rest.
Med Fragile parsing of structured data + divergent permission parsers
  • migrate.ts:603-616 & permissions.ts:291-340 regex-parse agents.yaml despite the YAML parser being imported (breaks on non-2-space indent / quoted values).
  • discover.ts:1106-1149 regex-scrapes openclaw's human-formatted table for session metadata — any cosmetic output change yields wrong data (then swallowed).
  • Two divergent JSONC strippers (permissions.ts:789 vs :1525) and two canonical-permission regexes (:465 vs :621) — a rule registered for one agent is silently dropped for another.
Fix: use yaml.parse and openclaw --json; one JSONC parser and one permission-pattern parser shared across converters.
